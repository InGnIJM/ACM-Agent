RAG 系统流程详解

RAG 是让大语言模型能够查找外部知识库的技术。下面我们用一道算法题的实际旅程，把整个流程讲清楚。适合 RAG 初学者阅读，每个概念都配了具体例子。

一、什么是 RAG？

RAG = Retrieval-Augmented Generation（检索增强生成）。 简单说：不让 LLM 凭空回答，而是先从一个知识库里搜出相关资料，再把资料连同问题一起交给 LLM 生成答案。

在这个项目中，知识库是 5 个 OJ 平台（Codeforces, AtCoder, LeetCode, Luogu, NowCoder）上的数万道算法题。用户可以用自然语言搜索——比如 "适合初学者的动态规划入门题" ——系统会找到语义最匹配的题目。

生活类比

就像考试时你可以翻课本——问题是你没见过的，但课本里有相关的知识点和例题。你翻书找到相关内容（Retrieval），然后结合自己理解写出答案（Generation）。LLM 自己没刷过所有算法题，但通过 RAG 它可以"翻书"。

RAG 的两个阶段

阶段一

离线录入 Offline

题目爬取 → LLM 摘要 → 向量嵌入 → 存入数据库。这个过程不面向用户，在后台批量完成。

阶段二

在线查询 Online

用户输入 → 向量化 → 三路召回 → 重排序 → 返回结果。毫秒级响应。

2

LLM 生成结构化摘要

爬下来的原始题目是 HTML 格式的英文或中文描述，不适合直接做语义搜索——有的题面 2000 字，有的只有 200 字，语言还不统一。系统调用 DeepSeek API 为每道题自动生成以下几种文本：

solution_summary

面向人类阅读的完整中文摘要 (300-500字)

用于前端展示 + 生成 solution 向量做语义检索

retrieval_summary

为向量搜索优化的精炼摘要 (150-350字)

专门用于生成 content 向量，跨题目相似度匹配

sparse_text

以空格分隔的关键词列表字符串

用于 PostgreSQL 全文搜索 (tsvector)，精确匹配算法术语

实际例子: LeetCode 1 "两数之和" → DeepSeek 生成的摘要

solution_summary (展示用)

给定一个整数数组 nums 和一个目标值 target，要求找出数组中和为目标值的两个数字的下标。通过维护一个哈希表记录已遍历元素的值和下标的映射，在遍历时检查 target - nums[i] 是否已存在于哈希表中，实现 O(n) 时间复杂度。关键点：哈希表同时承担了存储和查找的双重职责。常见陷阱：不能重复使用同一个元素，需要保证两个下标不同。

retrieval_summary (检索用)

题目：数组中两数之和等于目标值，返回下标。解法：哈希表一次遍历，存储已访问元素值和下标的映射，检查当前元素补数是否在哈希表中。算法：哈希表、数组遍历。时间复杂度 O(n)，空间复杂度 O(n)。

注意 retrieval_summary 比 solution_summary 更精炼、更多关键词、更少冗余描述——这就是为什么要分开：展示给用户看要详细，给向量搜索用要聚焦关键信息。

标签标准化

不同平台的标签体系不同（Luogu叫"动态规划dp"，LeetCode叫"Dynamic Programming"），系统用 taxonomy.json 映射到统一的 200+ 标签词汇表。

难度标准化

CF 的 800-3500、Luogu 的 1-7、LeetCode 的 E/M/H 全部映射到统一的 1-10 难度体系。

算法分类

LLM 还输出 primaryAlgo（主算法）、subAlgos（辅助算法）、problemPatterns（问题模式），用于训练 Agent 的个性化推荐。

3

文本转向量 (Embedding)

摘要文本通过嵌入模型转换成 1024 维的浮点数向量。一句话理解：嵌入模型把"意思"编码成数学坐标——意思相近的文本，向量在空间中距离就近。

E

Qwen3-Embedding-0.6B (llama.cpp)

llama-server · 端口 8089 · Q8_0 量化 · CUDA 加速 · Python 摄录管道和 NestJS 后端共用

关键设计: 指令感知嵌入 (Instruction-aware Embedding)

嵌入时给模型加一段"指令前缀"，告诉模型这段文本在系统中的角色。同样的文本在不同场景下产生的向量不同：

存储 content: "为算法题题面生成用于题意相似检索的向量" + [检索摘要文本]
存储 solution: "为算法题解法摘要生成用于相似题检索的向量" + [解法摘要文本]
查询: "为用户的算法题检索请求生成向量" + [用户查询文本]

为什么要这样？因为"两数之和用哈希表解决"这句话，出现在题目描述中（作为答案）和出现在用户查询中（作为提问）的语义角色完全不同。指令前缀帮助嵌入模型意识到这一点。

4

向量存储与索引

使用 PostgreSQL + pgvector 扩展存储。每道题的 3 个向量列分别建了 HNSW 索引，关键词列建了 GIN 全文搜索索引。

数据库向量列一览

列名内容来源索引类型
vector_embeddingsolution_summaryHNSW (cosine)
content_vectorretrieval_summaryHNSW (cosine)
summary_vectorfull_content (题面全文)HNSW (cosine)
sparse_text关键词列表GIN (全文搜索)

HNSW 索引原理 (类比理解)

把每道题想象成空间中的一个点。两题"意思相近"，在向量空间中距离就近。HNSW 索引就像预先建好的导航路线——查询时不用遍历所有点，沿索引路径就能快速找到最近邻居。

m=16 ef_construction=64 cosine 距离

为什么用 pgvector 而不是专用向量库？

题目数据本身就是结构化的（有标签、难度、平台等字段），需要 SQL 过滤 + 向量搜索同时进行。pgvector 把向量存在 PostgreSQL 里，一条 SQL 就能完成 "向量相似 + 平台=CF + 难度 3-7" 的复合查询。

2

三路并行召回 (Hybrid Recall) —— 核心设计

查询向量同时走三条通路，并行执行，3 秒超时保护。任意一条通路失败不影响其他路。每路独立返回 Top-K 结果。

C

content 向量检索

ANN 搜索 content_vector 列

→ 返回 Top-80

适合：题目描述的语义相似匹配

S

solution 向量检索

ANN 搜索 vector_embedding 列

→ 返回 Top-80

适合：解法和算法思路相似匹配

K

关键词全文搜索

ts_rank 搜索 sparse_text 列

→ 返回 Top-50

适合：精确算法名/术语匹配

为什么必须三条通路？各有所长

向量检索 (C + S)：擅长语义理解。"最短路径"能找到"Dijkstra"的题，即使题目里没有"Dijkstra"这个词。数学上，向量空间中相似概念天然聚集。但它不擅长精确术语匹配——"01背包"可能和"完全背包"混淆。

关键词搜索 (K)：擅长精确匹配。"01背包"直接命中标签为"01背包"的题目，不会混入完全背包。但"最优子结构"这种描述性语言搜不到任何东西——因为 sparse_text 里存的是关键词，不是自然语言描述。

三条通路取交集，既保证了语义覆盖率（向量通路），又保证了精确命中率（关键词通路）。这就是 "混合检索" 优于纯向量检索的原因。

3

合并去重 + Min-Max 归一化 + 动态加权

三条通路返回的分数尺度不同（向量用余弦距离，关键词用 ts_rank），需要统一尺度才能公平比较。系统使用 Min-Max 归一化将所有分数映射到 [0, 1] 区间，然后根据查询意图的权重计算最终粗分。

归一化与加权公式

score_norm = (score - min_score) / (max_score - min_score)
roughScore = w1 * score_content + w2 * score_solution + w3 * score_keyword

去重后约 150-200 条候选，按 roughScore 降序取 Top-50 送入重排序。

具体例子 (algorithm_intent)

weight = {content: 0.20, solution: 0.55, keyword: 0.25}

CF 1081D 背包 roughScore = 0.20*0.62 + 0.55*0.91 + 0.25*0.88 = 0.83
Luogu P1048 采药 roughScore = 0.20*0.55 + 0.55*0.87 + 0.25*0.85 = 0.79
AtCoder abc032_d roughScore = 0.20*0.48 + 0.55*0.82 + 0.25*0.65 = 0.71

CF 1081D 虽然是 solution 匹配最高的，但因为 keyword 权重偏低，总体仍然排第一。

4

LLM 重排序 (Reranking)

粗排序只看向量距离，可能遗漏语义细节。重排序阶段把 Top-20 候选的标题、摘要、标签一起发给 Qwen3-Reranker-0.6B 模型，让它逐对评估每道题与查询的语义相关性，而不是简单的向量距离。

举个典型场景：查询 "two pointers"，粗排可能把包含 "two" 和 "pointer" 两个单词的题目排得很高（关键词通路的字面匹配），但重排序会判断这些题是否真的用了双指针算法。"CF 279B Books" 虽然不含 "two pointers" 关键词，但解法确实是双指针，重排序后会被提升。

容错设计: 双重保障

如果重排序服务器不可用 → 自动回退到粗排序结果。
如果重排序分数方差 < 0.01（退化/所有题目分数一样） → 自动回退到粗排序结果。
用户始终能搜到东西，只是质量可能略降。

重排序前后对比 (查询: "two pointers 滑动窗口")

题目粗排重排后
CF 279B Books (双指针) 3 1 ↑
Luogu P1102 (含"two"关键词但非双指针) 1 3 ↓
LeetCode 3 (滑动窗口+哈希) 2 2 -

Luogu P1102 因为标题含 "two" 关键词被粗排提到第一，但重排序发现它跟双指针算法无关，被降级。CF 279B 虽然关键词匹配弱，但语义上确实是双指针，重排序后升到第一。这就是重排序的价值。

六、RAG 初学者的几个关键概念

多向量表示 (Multi-Vector)

为什么不为整道题生成一个向量？因为题目描述（"给你一个数组..."）和解题思路（"用哈希表 O(n)"）的语言风格完全不同，在向量空间中偏向不同方向。分开嵌入可以在不同维度上匹配用户意图——用户搜索"哈希表"时 solution 向量通路权重更高，搜索"数组处理"时 content 向量通路权重更高。

混合检索 (Hybrid Search)

纯向量检索对"01背包"这类精确术语不敏感——它会把所有含"0""1""背包"的向量都拉近，但不如直接全文搜索精确。纯关键词搜索对"寻找数组中两个数的和"这种自然语言描述完全无效。两者互补，取各自优势。

两阶段检索 (Two-Stage Retrieval)

向量检索速度快（ANN 索引，毫秒级）但粗糙（只看向量方向）。重排序慢但精细（逐字理解语义，百毫秒级）。先用向量快速从万级候选中筛出 Top-N，再用重排序精排——这种"粗筛+精排"是 RAG 系统的经典架构。

指令感知嵌入

同一段文本 "两数之和用哈希表解决"，在"作为题目存储"和"作为用户查询"时嵌入结果不同——因为任务目标不同。在文本前加指令前缀，告诉嵌入模型这段话在系统中的角色，可以显著提高检索精度。这是近两年嵌入模型的标配功能。

七、RAG 如何服务于训练 Agent

RAG 系统不仅是前端搜索框的后台，更是 训练 Agent (TrainingAgent) 的核心基础设施。这是一个基于 LangGraph 的智能体，为每个用户自动生成个性化周训练计划。

Agent 工作流: 加载用户画像 → 确定训练阶段 → 选择目标算法 → 计算难度曲线 → 调用向量搜索获取候选题目 → LLM 编排计划 → 保存
